# ML Project Documentation Generator — Reusable Prompt

## How to Use

Copy the prompt below, fill in `[PROJECT TYPE]` and `[PROJECT NAME]`, and send it to an AI assistant. You will receive a complete, production-grade ML project document matching the depth, structure, and quality of the three projects in your portfolio.

---

## The Prompt

---

You are an expert ML engineer and technical writer. Generate a **complete, production-grade ML project documentation** for the following:

- **Project Name**: [PROJECT NAME]  
  *(e.g., "Customer Churn Prediction", "Fraud Detection System", "Medical Image Classifier")*

- **Project Type**: [PROJECT TYPE]  
  *(e.g., "Binary Classification", "NLP / Text Classification", "Computer Vision / Object Detection", "Reinforcement Learning", "Graph ML", "Generative AI / RAG")*

---

### Output Requirements

Generate a single Markdown document that follows **exactly** this structure and depth. Each section must be fully written — no placeholders, no "TODO" comments. All code must be complete and executable.

---

### Document Structure to Follow

#### Header
- Portfolio-level one-liner introducing the project
- A summary table: Project | Type | Domain | Core Techniques

#### Section 1 — Problem Overview
- **Business Context**: 2–3 paragraph narrative of a realistic company, scale, and pain point with quantified business impact (revenue loss, cost, risk)
- **Objective**: One clear sentence stating what the ML system must achieve, with specific success metrics (e.g., "reduce churn by ≥15%")
- **Input → Output Definition**: A table with columns: Component | Detail — covering Inputs, Output, Latency SLA

#### Section 2 — Dataset
- **Dataset Sources & Links**: A table with real, verified Kaggle/HuggingFace/UCI datasets appropriate for this problem type. Columns: Component | What You Get | Link | License. Include 4–6 real dataset sources.
- **Dataset Assembly Strategy**: A diagram showing `FINAL DATASET = [components]` explaining what is real vs. synthetic and why
- **4-Step Dataset Assembly Code**: Complete Python code for:
  - Step 1: Download and unify real datasets (with exact column rename logic)
  - Step 2: Extract any calibration statistics from real data
  - Step 3: Generate synthetic data seeded with real IDs/distributions
  - Step 4: Merge into final unified dataset with summary print statement
- **Why This Combination Works**: A table explaining each table's origin (Real/Synthetic) and justification
- **Synthetic Data Generator**: Full, self-contained Python class with realistic distributions (power-law where appropriate, domain-specific correlations, temporal patterns). Include `if __name__ == "__main__"` runner.
- **Schema**: Column-by-column table for each CSV/table in the dataset
- **Sample Data**: 5-row sample of the most important table

#### Section 3 — System Design
- A full Mermaid diagram showing: data ingestion → feature pipeline → training pipeline → model registry → serving (real-time and/or batch) → monitoring

#### Section 4 — Data Pipeline
Three subsections, each with complete code:
- **Data Loading**: Full class with methods to load from the primary data source (BigQuery / Snowflake / InfluxDB / S3 — match what's realistic for this domain)
- **Data Cleaning**: Full class with methods for deduplication, outlier handling, and filtering. Include `WHY` comments on each major decision.
- **EDA**: A `run_eda()` function that prints key insights. Follow with 3–4 bullet "Key Insights" from the EDA.

#### Section 5 — Feature Engineering
Full class with 3–5 methods, each with a detailed docstring explaining `WHY` that feature matters for this specific problem (not generic ML platitudes — domain-specific reasoning).

#### Section 6 — Baseline Model
Implement the simplest possible baseline (heuristic or rule-based) with complete code. State its metric score.

#### Section 7 — ML Models
Implement **3 different model approaches** appropriate to the problem type. For each:
- Full training function with MLflow logging
- `WHY` this model is appropriate for this problem
- Expected metric score

#### Section 8 — Evaluation
- Define 3–4 metrics appropriate for this problem type (not just accuracy)
- Full evaluation function for each metric with docstring explaining `WHY` each metric is chosen over simpler alternatives
- Metric Interpretation section explaining what each number means in business terms

#### Section 9 — Model Comparison Table
Table with columns: Model | [Primary Metric] | [Secondary Metric] | [Tertiary Metric] | Training Time
- Include all 3 models + baseline
- Bold the winner row
- 2-paragraph explanation of why the winner was selected

#### Section 10 — Hyperparameter Tuning
Full Optuna objective function for the winning model. Include:
- All key hyperparameters with reasonable search ranges
- Best found values in a comment
- Best metric score achieved

#### Section 11 — Final Model & Predictions
- Complete inference function with fallback logic (handle edge cases: cold-start / missing features / unknown IDs)
- 5–10 line sample output showing what the model actually returns

#### Section 12 — Production Considerations
Three subsections with complete code:
- **Deployment**: FastAPI endpoint (for real-time) OR Airflow DAG (for batch) — choose based on the project's Latency SLA. Include Redis caching if real-time.
- **Scalability**: 4–5 bullet points on horizontal scaling, caching, partitioning strategies specific to this use case
- **Monitoring**: Complete monitoring class using Evidently or custom logic. Define 3 specific alert rules with thresholds and actions.

#### Section 13 — Project Folder Structure
Complete annotated folder tree with every file and a comment explaining its purpose. Include:
- `data/` (raw, processed, features)
- `notebooks/` (3–4 notebooks with realistic names)
- `src/` (data, features, models, evaluation, api or streaming)
- `airflow/dags/` (if batch)
- `monitoring/`
- `tests/`
- Root config files: `.env`, `Dockerfile`, `docker-compose.yml`, `requirements.txt`, `Makefile`, `README.md`

#### Section 14 — Deployment Addendum (REQUIRED)

**A. Fully Pinned requirements.txt**
Group by category with inline comments. Include all production dependencies with exact version numbers. Also provide a `requirements-dev.txt`.

**B. .env.example**
All environment variables the project needs, with realistic placeholder values and inline comments explaining each.

**C. Dockerfile**
Complete, production-ready Dockerfile. Include any system deps (gcc, libgomp, etc.) needed for ML libraries. Set `PYTHONPATH`, `EXPOSE`, and appropriate `CMD`.

**D. docker-compose.yml**
Full docker-compose with all required services (app + any of: Redis, Kafka/Zookeeper, InfluxDB, PostgreSQL, MLflow, Grafana). Each service must have correct image versions, environment variables, volumes, and `depends_on`.

**E. Step-by-Step: How to Create and Run This Project**
Numbered steps from `git clone` to a running, tested system. Each step must include:
- Exact bash commands (copy-paste ready)
- Expected output or runtime estimate
- What to verify before proceeding to the next step
Minimum 10 steps. Cover: environment setup → dataset download → data generation → infrastructure setup → training → evaluation → serving → monitoring.

**F. Common Issues & Fixes Table**
Table with columns: Issue | Cause | Fix. Include 6–8 realistic issues that developers actually encounter with the specific libraries used in this project.

#### Section 15 — Key Code Snippets
One end-to-end function that chains together the full pipeline (load → clean → features → predict → log → alert). This should be the "glue" function you'd actually call in production.

#### Section 16 — Extensions
4–5 concrete extension ideas specific to this project. Each must name a specific technique or algorithm (not generic "improve accuracy").

#### Section 17 — Resume-Ready Points
4 bullet points formatted for a résumé or LinkedIn. Each must:
- Start with a strong action verb
- Include specific metrics (latency numbers, percentage improvements, scale — users/rows/throughput)
- Reference specific techniques by name
- Connect to a quantified business impact ($ value, %, time saved)

---

### Quality Standards

Every piece of code must be **complete and executable** — no ellipsis (`...`), no `# implement this`, no stub functions. If a function is long, write the whole thing.

All `WHY` comments must be domain-specific. "This helps the model learn" is not acceptable. "We use kurtosis here because bearing faults create impulsive vibration signals that deviate strongly from Gaussian, and kurtosis directly quantifies this departure" is acceptable.

Metrics must be justified for the specific problem, not just listed. Explain why NDCG is better than accuracy for ranking, why AUC-PR is better than AUC-ROC for imbalanced data, etc.

Dataset links must be real, verified, and currently accessible (Kaggle, HuggingFace, UCI, government open data). Do not invent dataset URLs.

Business context must be quantified. "Reduces costs" is not acceptable. "Reduces annual inventory carrying costs by an estimated $1.4M based on 3.2% overstock rate across 500 SKUs" is acceptable.
